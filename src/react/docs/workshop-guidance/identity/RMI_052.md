# Planejar Recuperação de Conta com Microsoft Entra Verified ID

**Esforço de Implementação:** Médio – Requer coordenação entre as equipes de identidade, segurança e suporte técnico para selecionar um provedor de verificação de identidade na Microsoft Security Store, configurar políticas de recuperação de conta no centro de administração do Microsoft Entra, definir grupos de escopo de usuários e validar o fluxo de ponta a ponta antes de passar do modo de avaliação para o modo de produção.

**Impacto ao Usuário:** Médio – Os usuários que perdem todos os métodos de autenticação obtêm um caminho de autoatendimento para recuperar o acesso por meio de comprovação de identidade, reduzindo a dependência do suporte técnico. Durante a implantação, os grupos de usuários no escopo devem ser informados sobre o novo fluxo de recuperação e pode ser necessário instalar ou atualizar o Microsoft Authenticator.

## Visão Geral

Quando um usuário perde acesso a todos os métodos de autenticação registrados — um dispositivo perdido, códigos de backup indisponíveis, nenhum telefone registrado — a redefinição de senha por autoatendimento tradicional não pode ajudar porque requer pelo menos um fator funcionando. Em ambientes sem senha, esse problema é amplificado: não há senha para recorrer, e o único caminho de recuperação é uma chamada ao suporte técnico, onde um agente deve verificar manualmente a identidade do chamador antes de emitir novas credenciais. Essa etapa de verificação manual é um dos vetores mais explorados em ataques de identidade. Os agentes de ameaças rotineiramente usam engenharia social para convencer os agentes de suporte a redefinir credenciais de contas que não lhes pertencem, contornando todos os controles upstream que a organização implantou.

A recuperação de conta do Microsoft Entra ID aborda essa lacuna substituindo o julgamento humano na etapa de verificação por comprovação de identidade criptográfica. O fluxo funciona redirecionando um usuário bloqueado para um provedor de verificação de identidade (IDV) confiável de terceiros — como Idemia, LexisNexis ou Au10tix — disponível na Microsoft Security Store. O IDV valida o documento de identificação emitido pelo governo do usuário, realiza detecção biométrica de vivacidade e emite uma credencial verificável na carteira Microsoft Authenticator do usuário via Microsoft Entra Verified ID. O usuário então apresenta essa credencial ao Microsoft Entra ID, que a valida criptograficamente, compara os atributos de identidade com o perfil de usuário armazenado e emite uma Senha de Acesso Temporário para que o usuário possa se cadastrar novamente em novos métodos de autenticação.

Essa abordagem suporta diretamente o princípio de Zero Trust de *Verificação Explícita* ao substituir verificações informais de identidade pelo suporte técnico por comprovação com base em documentos e verificação biométrica que produz uma credencial à prova de adulteração. Suporta o *Assumir Violação* ao remover completamente a superfície de ataque de engenharia social do processo de recuperação — um agente de ameaças que comprometeu os dispositivos de um usuário ainda não consegue passar pela verificação de documentos e biometria pelo IDV. Também reforça o *Acesso com Menor Privilégio* ao limitar a recuperação a grupos de usuários específicos e emitir apenas uma Senha de Acesso Temporário com validade limitada, em vez de credenciais permanentes, garantindo que o direito de recuperação seja restrito e de curta duração. Sem essa funcionalidade, as organizações em ambientes sem senha permanecem dependentes de recuperação mediada pelo suporte técnico, o que introduz tanto um gargalo operacional quanto um risco persistente de engenharia social que prejudica a postura de segurança de todos os controles de autenticação upstream.
